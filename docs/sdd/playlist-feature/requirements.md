# 要件定義書 — プレイリスト機能

## 概要

AI Music Station にユーザーが自分専用のプレイリストを作成・管理し、お気に入りトラックをまとめて連続再生できる機能を追加する。

## 背景

現状の AI Music Station はチャンネル単位の「ラジオモード」のみで、ユーザーが好きなトラックを選んで順序立てて聴くことができない。お気に入り（リアクション）機能はあるが、それらをまとめて再生する手段が存在しない。プレイリスト機能を追加することで、ユーザーの能動的な音楽体験と滞在時間の向上を図る。

## 機能要件

| ID | 要件 | 優先度 | 備考 |
|----|------|--------|------|
| FR-001 | ユーザーはプレイリストを作成できる（名前・説明付き） | Must | セッションID単位で管理 |
| FR-002 | ユーザーはプレイリストを編集できる（名前・説明の変更） | Must | |
| FR-003 | ユーザーはプレイリストを削除できる | Must | 論理削除 |
| FR-004 | ユーザーはトラックをプレイリストに追加できる | Must | 重複追加は不可 |
| FR-005 | ユーザーはプレイリストからトラックを削除できる | Must | |
| FR-006 | ユーザーはプレイリスト内のトラック順序を並べ替えできる | Must | ドラッグ&ドロップ |
| FR-007 | ユーザーはプレイリストを順番に再生できる | Must | 既存 Player コンポーネントを流用 |
| FR-008 | お気に入り（リアクション済み）トラックを一覧表示できる | Should | 既存の reaction データを活用 |
| FR-009 | お気に入りトラック一覧からプレイリストに追加できる | Should | |
| FR-010 | プレイリスト再生中に「次のトラック」「前のトラック」操作ができる | Must | |
| FR-011 | プレイリストはセッションをまたいで保持される | Must | localStorage + サーバー保存 |
| FR-012 | プレイリスト一覧を表示できる（最大50件） | Must | |

## スコープ

### In（対象範囲）

- プレイリスト CRUD（作成・読取・更新・削除）
- プレイリストへのトラック追加・削除・並べ替え
- プレイリスト再生（順番再生・スキップ・前へ）
- お気に入りトラック一覧表示とプレイリストへの追加
- DB スキーマ追加（Alembic マイグレーション）
- REST API 追加（FastAPI）
- フロントエンド UI 追加（React + TypeScript）

### Out（対象外）

- プレイリストの公開・共有機能（将来拡張）
- プレイリストへのコラボレーション（複数ユーザー編集）
- シャッフル再生モード（将来拡張）
- プレイリストのエクスポート（M3U等）
- ユーザー認証・アカウント管理（session_id ベースのまま）
- Liquidsoap ストリームへのプレイリスト統合

## 用語定義

| 用語 | 定義 |
|------|------|
| プレイリスト | ユーザーが任意のトラックを任意の順序で並べたコレクション |
| セッションID | ブラウザ側で生成・localStorage 保持する UUID。ユーザー識別子として使用 |
| トラック | AI 生成された音楽ファイル（Track テーブルの1レコード） |
| お気に入り | reaction_type="like" のリアクション済みトラック |
| 並べ替え | プレイリスト内のトラック表示順序（position カラム）の変更 |
| ラジオモード | 既存のチャンネルストリーミング再生（本機能とは独立） |

## リスク評価

| ID | リスク | 影響度 | 発生確率 | 対策 |
|----|--------|--------|----------|------|
| R-001 | 既存 Player コンポーネントとの再生モード競合 | 高 | 中 | 再生ソース（radio/playlist）を明確に分離 |
| R-002 | セッションIDが変わるとプレイリストにアクセス不可 | 中 | 低 | localStorage で session_id を永続化 |
| R-003 | 並べ替えの position 更新で DB 負荷増大 | 低 | 低 | バルクアップデートで1リクエストに集約 |
| R-004 | 大量プレイリスト（50件×200曲）の取得パフォーマンス | 中 | 低 | ページネーション・遅延読み込み |

## ロールバック方針

- 切り戻し条件: プレイリスト API エラー率 > 5%、または既存ラジオ機能の障害が発生した場合
- 切り戻し手順:
  1. 新規 Alembic マイグレーションで `playlists`・`playlist_tracks` テーブルを DROP
  2. フロントエンドのプレイリスト関連コンポーネントを feature flag で非表示
  3. 既存テーブル・API への変更は一切ないため、ラジオ機能は即時復旧
- 影響範囲: プレイリストデータは失われるが、既存トラック・チャンネル・リアクションデータへの影響なし

## 受入基準（AC）

| AC-ID | 条件 | 確認方法 |
|-------|------|----------|
| AC-001 | プレイリストを新規作成すると DB に1レコード挿入され、レスポンスに UUID が返る | API テスト（POST /api/playlists） |
| AC-002 | 同一 session_id で最大50件のプレイリストが作成できる | E2E テスト |
| AC-003 | プレイリストに同一トラックを2回追加しようとすると 409 エラーが返る | API テスト |
| AC-004 | トラック並べ替え後、GET で正しい position 順に返る | API テスト |
| AC-005 | プレイリスト再生時、Player が playlist モードで各トラックを順番に再生する | E2E テスト |
| AC-006 | 別セッションIDからのプレイリスト操作は 403 エラーが返る | API テスト |
| AC-007 | プレイリスト削除後、関連 playlist_tracks も全て削除される（CASCADE） | DB 確認 |
| AC-008 | 既存のラジオ機能（チャンネルストリーミング）が影響を受けない | E2E テスト |
| AC-009 | お気に入りトラック一覧に reaction_type="like" のトラックが表示される | E2E テスト |
| AC-010 | モバイル（375px幅）でプレイリスト管理画面が正常に表示・操作できる | Playwright スクリーンショット |
